# Heat Sheet Generator: Design and Implementation Change Log

## Date
- 2026-06-30

## Scope
This log summarizes recent design and implementation updates made to improve performance, add prelim-specific seeding behavior, and make generation behavior more predictable.

## 1) Prelim Circle Seeding Design

### What changed
- Added prelim event detection in seeding logic.
- If an event name includes prelim or preliminary, and circle seeding is enabled, the fastest 3 heats use cross-heat circle seeding.
- For those top 3 heats, the top 3 seeds are assigned to lane 4 across the last 3 heats (fastest heat first).

### Why
- Matches requested prelim behavior for championship-style psych sheets.

### Current behavior
- Prelim + enabled: fastest 3 heats use cross-heat circle seeding.
- Not prelim or disabled: standard per-heat lane assignment remains unchanged.

## 2) Circle Seeding Enable/Disable Control

### What changed
- Added a sidebar toggle: Enable Prelim Circle Seeding.
- Toggle state is passed into meet settings and checked by seeding logic.

### Why
- Allows turning prelim-specific behavior on or off without code edits.

### Current behavior
- Enabled: prelim circle logic applies.
- Disabled: all events use standard behavior.

## 3) Streamlit Generation Performance Refactor

### What changed
- Replaced one combined generation key with two keys:
  - Layout key: file + lane config + long-distance order + prelim circle toggle.
  - Favorites key: selected favorite swimmers.
- Reseeding now runs only when layout settings change.
- Output regeneration runs when either layout or favorites change.

### Why
- Avoids unnecessary reseeding and expensive recomputation when only favorites change.
- Fixes stale-output risk when prelim toggle changes.

### Current behavior
- Layout-only change: reseed + regenerate outputs.
- Favorites-only change: no reseed; regenerate outputs that depend on favorites.

## 4) Favorite PDF Generation Optimization

### What changed
- Favorite swimmers PDF generation is now lazy.
- It is generated only when the user clicks the favorite PDF download button.
- Generated bytes are cached in session state for reuse.

### Why
- Removes unnecessary PDF work from the main Generate action.
- Improves responsiveness for users who do not download favorite PDFs.

## 5) HTML Preview Optimization

### What changed
- In HTML preview generation, favorite swimmer names and favorite event numbers are precomputed as sets.
- Event-level favorite expansion and row highlighting now use set membership checks.

### Why
- Reduces repeated nested scanning and improves rendering performance for larger meets.

## 6) Models Summary

### Core model objects
- Meet
  - Top-level container for one meet.
  - Holds name, settings, and all events.
  - Provides helpers to iterate entries and derive favorites.
- MeetSettings
  - Runtime seeding controls (lanes, circle seed config, distance event order, long-distance set).
- Event
  - Represents one event (event number + name).
  - Stores parsed entries and generated heats.
- Entry
  - A swimmer's entry in one event.
  - Stores seed number/time and generated heat/lane assignment.
  - Keeps back-reference to its event.
- Swimmer
  - Identity fields for display and grouping (name, gender, age, team).
- Heat
  - One heat in an event with heat number and lane list.
- Lane
  - Lane number with optional entry assignment.

### Relationship map
- Meet -> many Event
- Event -> many Entry (input psych sheet)
- Event -> many Heat (generated output)
- Heat -> many Lane
- Lane -> optional Entry
- Entry -> one Swimmer

### Data flow (implementation view)
1. Parser reads psych sheet text and creates Event plus Entry plus Swimmer objects.
2. Seeding transforms Event.entries into Event.heats and fills Entry.heat_number and Entry.lane_number.
3. Exporters consume the same model graph:
   - HTML exporter reads events/heats/lanes and favorite mappings.
   - PDF exporter reads events/heats/lanes and favorite mappings.

### Notes on behavior
- Entry is the bridge between parse-time data and seeded output.
- Favorite output is computed from Meet.favorite_entries, keyed by swimmer name.
- Current implementation assumes stable parsed ordering from the psych sheet input.

## Notes
- Existing assumptions retained: psych sheet order is already seed-time sorted.
- No parser-name logic was changed based on current user constraints.
